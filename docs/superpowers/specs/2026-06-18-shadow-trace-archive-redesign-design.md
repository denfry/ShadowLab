# Shadow Trace — пересмотр концепта прохождения: «Архив-расследование»

- **Дата:** 2026-06-18
- **Статус:** дизайн утверждён, готов к плану реализации
- **Заменяет:** прохождение из `2026-06-18-shadow-trace-v2-design.md` (петля «стол следователя + доска связей»). Этот документ переопределяет **только** концепт прохождения и движок навигации; пилларсы «честной» сложности и «поймать на лжи» сохраняются.

---

## 1. Контекст и мотивация

Текущий v2-концепт — «стол следователя» с шестью вкладками и **доской связей**, где игрок руками соединяет факты, а движок матчит авторские противоречия. Решение по итогам брейншторма: петля «доска + противоречия» в корне не подходит — нужен другой жанр прохождения.

**Новое направление — архив-расследование** (ориентир: Her Story / Obscura), но с навигацией через **связанный поиск по сущностям-гиперссылкам**, а не свободный ввод слов.

Что сохраняем из v2:
- «Честная» сложность: без таймеров и RNG, трудность — в синтезе фактов.
- «Поймать на лжи» как главный глагол.
- Несколько защитимых концовок и ранговый скоринг.
- Процедурное медиа (SVG/CSS), уже реализованный фото-инспектор с хотспотами/Enhance/метаданными.

Что выкидываем:
- Граф «зацепок-нод» (lead nodes) и условное гейтирование нод.
- Доску связей и ручное соединение фактов как механику прохождения.

## 2. Видение прохождения

**Метафора.** Игрок — аналитик за **терминалом архива дела**. Вся игра — одно рабочее пространство-браузер по материалам следствия (эстетика защищённого терминала / старой базы данных). Нет «фаз» и «ходов» — чтение и хождение по ниточкам.

**Петля.**
```
открыть запись → в тексте подсвечены сущности (имена/места/время/объекты)
  → кликнуть сущность → её карточка: все доступные записи о ней + связанные сущности
  → открыть новую запись → новые сущности → глубже
  → наткнулся на запечатанный файл → нашёл ключ/пароль в другой записи (или хотспотом на фото) → открыл
  → заметил, что два показания / факт+метаданные не сходятся → закрепил их в досье
  → набрал картину → «Заключение»: назвать виновного + ткнуть в решающую ложь
```

**Три «соли».**
1. **Идёшь по нитке, а не угадываешь слово.** Граф дела раскрывается кликами по сущностям; каждая ведёт к новым записям. Гейтинг — эмерджентный: запись достижима, как только открыта ведущая к ней сущность.
2. **Запечатанные файлы — точечные «аха».** Часть записей запечатана (шифр / уровень доступа / редактура); печать снимается ключом, найденным в другой записи или через хотспот на фото. Даёт темп без линейности.
3. **Ложь живёт в перекрёстных ссылках.** На карточке сущности рядом оказываются записи, которые друг другу противоречат. Игра не подсвечивает — игрок замечает сам и закрепляет. Это и есть «поймать на лжи», без отдельной доски.

**Условие победы.** Назвать виновного **и** указать решающую ложь — пару фактов (запись↔запись или запись↔метаданные), совпадающую с авторским противоречием.

## 3. Экран и UX

Единое пространство (не вкладки):

```
┌──────────────┬───────────────────────────────┬──────────────┐
│ ИНДЕКС        │ ЧИТАЛКА                        │ ДОСЬЕ        │
│ (открытое)    │ запись ИЛИ карточка сущности   │ (закладки)   │
│ Лица     ▸   │ ┌── Допрос Эрона, 12.06 ──┐    │ 📌 записи    │
│  Эрон        │ │ «Я был дома в 22:00...» │    │ 📌 сущности  │
│  Мара        │ │ упомянуты: [Эрон] [офис]│    │ ⚑ подозрения │
│ Места    ▸   │ │ [22:00] [Мара]          │    │ ✎ заметки    │
│ Время    ▸   │ │ 📎 фото (CCTV) →инспектор│    │              │
│ Объекты  ▸   │ └─────────────────────────┘    │ [ ЗАКЛЮЧЕНИЕ ]│
│ Записи   ▸   │ ← назад · хлебные крошки       │              │
└──────────────┴───────────────────────────────┴──────────────┘
```

- **Индекс (слева):** только уже открытое — сущности по типам со счётчиками записей + список записей. Фильтр по открытому (НЕ глобальный поиск по скрытому контенту).
- **Читалка (центр):** либо **запись** (текст с кликабельными сущностями + прикреплённое процедурное фото, открывающее существующий фото-инспектор), либо **карточка сущности** (сводка + все доступные записи о ней; запечатанные показаны с 🔒 и подсказкой; + связанные сущности). Хлебные крошки и «назад» для возврата по нитке.
- **Досье (справа):** закреплённые записи и сущности, помеченные подозрения, заметки, кнопка **Заключение**. Доски нет — закреплённое лежит рядом для сопоставления глазами.

## 4. Модель данных (контент дела)

Дело — `CaseArchive`, хранится как `public/data/archive-cases/<id>.json`.

```ts
interface CaseArchive {
  id: string;
  title: string;
  difficulty: 'normal' | 'hard' | 'nightmare';
  synopsis: string;
  episodeOf?: string;               // задел под кампанию (A2), в A0/A1 не используется
  seedRecordIds: string[];          // записи, открытые на старте
  records: Record[];
  entities: Entity[];
  contradictions: Contradiction[];  // «лжи» — переиспользованное понятие из v2
  endings: Ending[];
  keysSchema: KeyDef[];             // именованные ключи/пароли для печатей
}

interface Record {
  id: string;
  kind: 'transcript' | 'chat' | 'letter' | 'report' | 'log' | 'note' | 'photo' | 'object';
  title: string;
  source?: string;                  // напр. «Допрос, 12.06, каб. 4»
  timestamp?: string;
  body: RecordSpan[];               // текст с инлайновыми ссылками-сущностями
  mediaId?: string;                 // прикреплённое процедурное медиа (готовый MediaSpec)
  metadata?: { time?: string; geo?: string; device?: string; exif?: Record<string, string> };
  mentions: string[];               // id упомянутых сущностей (для валидатора; согласуется с body)
  grantsKeys?: string[];            // чтение записи (или её хотспот) выдаёт эти ключи
  seal?: { keyId: string; hint: string };  // запечатано, пока нет ключа keyId
}

type RecordSpan =
  | { text: string }
  | { entityId: string; text: string };     // кликабельная сущность

interface Entity {
  id: string;
  type: 'person' | 'place' | 'time' | 'object' | 'event' | 'org';
  label: string;
  aliases?: string[];
  summary?: string;                 // показывается на карточке сущности
  isSuspect?: boolean;              // можно ли обвинить
}

interface KeyDef { id: string; label: string; }

// FactRef расширяем, чтобы указывать на записи архива:
type FactRef =
  | { kind: 'record'; recordId: string }
  | { kind: 'recordClaim'; recordId: string; claimId?: string }
  | { kind: 'metadata'; recordId: string; field: string }
  | { kind: 'entity'; entityId: string };

interface Contradiction {
  id: string;
  between: [FactRef, FactRef];
  rule: 'time_overlap' | 'place_conflict' | 'mutual_exclusive' | 'authenticity';
  weight: number;
  revealHint?: string;              // опциональная подсказка для UI/скоринга
}
```

**Заметки по модели.**
- `mentions` дублирует сущности из `body`-спанов намеренно — это явный вход для валидатора достижимости и упрощает индексацию без парсинга спанов.
- `grantsKeys` на записи покрывает два источника ключей: чтение самой записи и срабатывание хотспота прикреплённого медиа (хотспот тоже выдаёт ключ через тот же механизм эффектов).
- Один-два `Record` с `authenticity`-ролью (подделка) выражаются через `Contradiction` с `rule: 'authenticity'` между записью и её метаданными.

## 5. Состояние игрока и движок

Движок — чистые иммутабельные функции (как Этап 0 v2). Состояние — простой объект, без Zustand на уровне логики.

```ts
interface ArchiveProgress {
  caseId: string;
  openRecords: string[];        // читаемые сейчас (seed + достигнутые и распечатанные)
  seenRecords: string[];        // прочитанные хоть раз
  discoveredEntities: string[]; // открытые сущности (попадают в индекс)
  keys: string[];               // полученные ключи (снимают печати)
  pinnedRecords: string[];
  pinnedEntities: string[];
  suspicions: { recordId: string; note?: string }[];
  notes: string[];
  accusation?: { culpritEntityId: string; decisiveLie?: [FactRef, FactRef] };
}
```

**Функции-переходы.**
- `createArchiveProgress(caseData): ArchiveProgress` — открывает `seedRecordIds`, заносит их упомянутые сущности в `discoveredEntities`, выдаёт `grantsKeys` seed-записей, авто-распечатывает то, что стало доступно.
- `openRecord(caseData, state, recordId): ArchiveProgress` — no-op, если запись недостижима или запечатана без ключа. Иначе: помечает `seen`, добавляет её сущности в `discoveredEntities`, выдаёт `grantsKeys`, авто-распечатывает записи, чьи `seal.keyId` теперь в `keys`, и пересчитывает `openRecords`.
- `pinRecord` / `pinEntity` / `unpinRecord` / `unpinEntity` — работа с досье (идемпотентны, дедуплицируют).
- `markSuspicion(state, recordId, note?)` / `clearSuspicion` — пометки подозрений.
- `addNote(state, text)` — свободная заметка.
- `accuse(caseData, state, accusation): ArchiveProgress` — записывает обвинение (резолв концовки/скоринга — через селекторы, см. §6–7).

**Правило достижимости.** Запись достижима, если: она в `seedRecordIds`, **или** упоминает хотя бы одну сущность из `discoveredEntities`. Достижимая запись попадает в `openRecords` только если у неё нет `seal` **или** `seal.keyId ∈ keys`. Запечатанные достижимые записи показываются в карточке сущности как 🔒 с `hint`, но тело не читается. Открыл запись → открылись её сущности → стали достижимы записи-упоминания → граф ширится сам.

## 6. Селекторы (read-only API для UI)

- `getRecordView(caseData, state, recordId)` → `{ record, spans: ResolvedSpan[], media?, metadata?, sealed: boolean, sealHint? }`. Резолвит entity-спаны в кликабельные ссылки; помечает, открыта ли запись.
- `getEntityPage(caseData, state, entityId)` → `{ entity, records: { record, sealed }[], relatedEntities: Entity[] }`. `records` — все записи, упоминающие сущность (запечатанные с флагом); `relatedEntities` — сущности, со-упомянутые в этих записях.
- `getDiscoveredIndex(caseData, state)` → сущности из `discoveredEntities`, сгруппированные по `type`, со счётчиком доступных записей; + список открытых записей.
- `getCaseFile(caseData, state)` → закреплённые записи/сущности, подозрения, заметки (резолвленные в отображаемые объекты).
- `getAccusableSuspects(caseData, state)` → открытые сущности `type:'person'` с `isSuspect`.
- `checkAccusation(caseData, state, accusation)` → `DeductionResultV2` + выбранная `Ending` (см. §7).

## 7. Концовки и скоринг (спайн переиспользуем)

**Резолв концовки** по правильности обвинения:
- `truth` — верный виновный **и** `decisiveLie` совпадает с авторским противоречием (порядок пары не важен).
- `partial` — верный виновный, но `decisiveLie` отсутствует/не совпала.
- `miscarriage` — обвинён не тот.
- `cold_case` — обвинения нет (фолбэк).

`Ending` сохраняет форму из v2 (`id`, `title`, `requires`, `quality`, `epilogue`, `campaignEffects?`), но `requires` оперирует фактами архива (виновный + найденные противоречия) — резолвер портируется под новую модель состояния.

**Производное «найдено противоречие».** В новой модели нет авто-открытия противоречий по ходу. Поэтому метрика выводится из досье: авторское `Contradiction` считается «замеченным», если **обе** записи из его `between` закреплены (`pinnedRecords`) или помечены подозрением (`suspicions`). Это даёт сигнал «игрок свёл рядом то, что не сходится», не возвращая доску.

**Скоринг `DeductionResultV2` (ранг F–S)** — порт из v2:
- найдено ли решающее противоречие (= `decisiveLie` совпала с авторским противоречием наибольшего веса);
- доля авторских противоречий, «замеченных» по правилу выше;
- доля вскрытых печатей;
- штраф за ложное обвинение (не тот виновный) и за пустые подозрения, не входящие ни в одно противоречие.
Маппинг в ранг: S ≥ 95, A ≥ 80, B ≥ 65, C ≥ 45, F < 45.

## 8. Валидатор (фикспойнт достижимости)

От `seedRecordIds` симулируем до неподвижной точки: открытие записей → открытие сущностей → достижимость записей-упоминаний → выдача `grantsKeys` → снятие печатей. Проверяем:
- каждая запись достижима из старта;
- каждый ключ из `keysSchema` добываем; каждая `seal.keyId` снимаема;
- оба `FactRef` каждого `Contradiction` указывают на достижимые записи/сущности;
- существует хотя бы одна `truth`-концовка и она достижима;
- нет висячих ссылок: `entityId` / `recordId` / `keyId` / `mediaId` существуют;
- нет дублей id (записи, сущности, ключи, противоречия, концовки);
- границы медиа (хотспоты/артефакты) валидны (порт из v2).

## 9. Дело-пример (вертикальный срез A1)

Скелет Эрон/Мара, переосмысленный под архив:
- **Seed-записи:** рапорт о происшествии (упоминает жертву, офис, время, Эрона, Мару) + допрос Эрона («был дома в 22:00»).
- **Сущности:** Эрон (person, suspect), Мара (person, suspect), офис (place), 21:30 / 22:00 (time), пропуск-карта (object), CCTV-кадр (event).
- **Запечатанная запись:** журнал доступа (log), `seal.keyId = 'k_admin'`; ключ `k_admin` выдаётся чтением чата администратора или хотспотом на CCTV-фото.
- **Решающая ложь:** показание Эрона (дома 22:00) ↔ метаданные журнала (карта в 21:28) / метаданные CCTV (21:30), `rule: 'time_overlap'`.
- **Концовки:** `truth` (Эрон + та самая пара), `partial` (Эрон, пара не та), `miscarriage` (Мара), `cold_case` (без обвинения).

## 10. Объём и этапы

Этот пересмотр заменяет старый Этап 1b. Новая разбивка:

- **Этап A0 — Архивный движок (чистая логика, без UI).** Новые типы (§4–5), машина навигации, логика открытия + печатей/ключей, портированные `matchContradiction` / резолв концовки / скоринг / валидатор. Полное юнит-покрытие. Зеркалит старый Этап 0.
- **Этап A1 — Вертикальный срез + одно дело.** Терминал архива (индекс / читалка / досье), ссылки-сущности, читалка записи, карточка сущности, поток снятия печати, закладки/подозрения, экран «Заключение» + концовка, переиспользование фото-инспектора. Один авторский `archive-case` JSON (§9). Проходится от начала до конца.

**Объём ЭТОГО спека = A0 + A1.**

**Отложено (вне спека, Этап A2 и далее):** богаче печати-через-медиа, второе дело, уровни сложности, кампания (`episodeOf`, перенос закреплённых фактов/флагов/репутации), «мотив» в обвинении.

## 11. Судьба старого кода

- Старый `src/games/shadow-trace/engine/` (граф зацепок, доска связей, селекторы доски) **выводится из эксплуатации**.
- Ценный спайн (`contradictions.ts`, `endings.ts`, скоринг, `validator.ts`, типы медиа) — **портируется** в новый модуль `src/games/shadow-trace/archive/`, без граф-нод.
- Старый `engine/` **парковать** (не удалять сразу) и снести после того, как срез A1 заработает и подтвердит паритет.

## 12. Тесты

- **Юнит:** навигация (открытие записей / находка сущностей / снятие печатей / выдача ключей), матч противоречий по записям, резолв концовки + скоринг, валидатор (достижимость, висячие ссылки, дубли id, границы медиа, наличие truth-пути).
- **Интеграция:** «пройти дело-пример» (seed → нитка → ключ → распечатать журнал → верное обвинение → truth/S); ветка «не тот виновный → miscarriage»; штраф за ложное подозрение.
- Раннер и конвенции — как у текущих 127 тестов.

## 13. Принципы (YAGNI)

- Обвинение в A1 = виновный + решающая ложь. Мотив/метод — отложены.
- Кампания смоделирована полем `episodeOf`, но логика переноса — A2.
- Свободного полнотекстового поиска по скрытому контенту нет: навигация только через сущности-ссылки и фильтр по открытому.
