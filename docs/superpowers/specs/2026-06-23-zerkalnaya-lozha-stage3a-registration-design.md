## 0. Контекст и решения

Портал монтирует игры через `GameModule`/`GameRegistry`. Shadow Trace — чисто-
React игра: `mount()` — no-op, вся игра живёт в `Hud: ({ctx}) => <Game ctx={ctx}/>`.
Лодж ляжет так же (R3F-`<Canvas>` — просто часть React-дерева `Hud`). На текущей
ветке Лодж доступен только через DEV-роут `/dev/lodge` (меню Соло/Мультиплеер).
Финальное ревью Этапа 2 нашло перекос: `chooseOwners` назначает lock/clue по
каждому пазлу независимо → бывает забег, где у одной роли все замки (хост ничего
не оперирует).

Решения брейншторма (2026-06-23):

- **Точка входа каталога — только мультиплеер:** игра = лобби (имя →
  создать/войти по коду) → кооп-игра. Соло остаётся dev-affordance на
  `/dev/lodge`.
- **Тему переиспользуем** (`theme: 'shadow'`) — тёмный/оккультный вайб подходит;
  своя тема и стилизация — отдельный полиш позже. Union `GameTheme` не трогаем.
- **Баланс:** детерминированный пере-флип одного пазла, если забег однобокий.

## Дизайн-столпы

1. **Переиспользование, не переписывание.** Каталожная игра — это `LobbyScreen`
   + `NetGameView` Этапов 1–2, обёрнутые в `Hud` с `ctx`. Минимум нового кода.
2. **Движок — источник истины (включая баланс).** Баланс ролей живёт в
   `createRun` (детерминированно), а не в UI; покрыт тестами.
3. **Изоляция правок.** Этап-2 файлы трогаем минимально и аддитивно
   (`LobbyScreen` получает опциональные пропы); dev-харнесс продолжает работать.
4. **Эфемерность сохраняется.** Партии не персистятся; `ctx.save`/achievements
   не используем (вне области).

## Секция 1 — Что строим и определение «готово»

**Часть A — регистрация (мультиплеер-only):**

- `GameId` += `'lodge'`. `LODGE_DEFINITION.theme = 'shadow'` (тему не добавляем).
- `LODGE_DEFINITION: GameDefinition` — id `lodge`, название, таглайн, описание,
  `status: 'available'`, теги, bootHint.
- `LodgeGameModule` — `mount` no-op, `Hud: ({ctx}) => <LodgeGame ctx={ctx}/>`,
  `payloadVersion: 1`.
- `LodgeGame` — точка входа: состояние `{lobby | net}`; `LobbyScreen` →
  `NetGameView`; `onExit` из net → обратно в лобби; «← В портал» → `ctx.exit()`;
  `ctx.params.room` предзаполняет код (шарящиеся ссылки «вступай по ссылке»).
- Регистрация в `src/games/index.ts`; запуск через существующий лаунчер
  `/play/lodge`. Dev-роут `/dev/lodge` (с Соло) остаётся.
- Минимальная аддитивная правка Этапа 2: `LobbyScreen` получает опциональные
  `initialCode?: string` и `onExitToPortal?: () => void`; в dev-харнессе они не
  передаются — поведение прежнее.

**Часть B — баланс ролей (`createRun`):**

- Хелпер `flipPuzzleRoles(p)` = `{ ...p, clueOwner: other(clueOwner),
  lockOwner: other(lockOwner), views: { A: p.views.B, B: p.views.A } }`
  (solution/state не трогаем; `lock≠clue` сохраняется).
- В `createRun`: после сборки цепочки, если `puzzles.length ≥ 2` и у всех один
  `lockOwner` — перевернуть `puzzles[0]`. Детерминированно. Гарантия: при ≥2
  пазлах у каждой роли ≥1 интерактивный замок.

**«Готово» для 3a:**

1. «Зеркальная Ложа» появляется в каталоге портала и запускается через
   `/play/lodge`.
2. Запуск показывает мультиплеер-лобби → кооп-игру (сеть Этапа 2).
3. «← В портал» возвращает в портал (`ctx.exit()`); опц. `?room=CODE`
   предзаполняет вход.
4. Генератор гарантирует обоим игрокам ≥1 замок (нет однобоких забегов).
5. Тесты зелёные; `npm run build` ок.

## Секция 2 — Архитектура, файлы, тесты

**Файлы:**

- `src/types/game-module.ts` — *modify*: `GameId` += `'lodge'`.
- `src/games/lodge/definition.ts` — *create*: `LODGE_DEFINITION`.
- `src/games/lodge/ui/LodgeGame.tsx` — *create*: каталожная точка входа
  (`lobby ↔ net`, ctx-aware: exit, params.room).
- `src/games/lodge/LodgeGameModule.tsx` — *create*: `lodgeModule: GameModule`.
- `src/games/index.ts` — *modify*: зарегистрировать lodge (ленивая загрузка).
- `src/games/lodge/dev/LobbyScreen.tsx` (Этап 2) — *modify, аддитивно*:
  опциональные `initialCode?`/`onExitToPortal?`.
- `src/games/lodge/engine/generate.ts` — *modify*: `flipPuzzleRoles` + балансная
  правка в `createRun`.

**Главный риск регистрации:** добавление `'lodge'` в union `GameId` может
«всплыть» в исчерпывающих `switch(gameId)` / маппингах по порталу — `tsc` это
поймает, и реализатор добавит ветку `lodge` там, где компилятор пожалуется
(тема/иконка/постер-карта/etc.). План отмечает это явно: после правки union
прогнать `npm run typecheck` и закрыть все всплывшие места минимально.

**ctx-вайринг (`LodgeGame`):** используем только `ctx.exit()` (назад в портал) и
`ctx.params.room` (опц. предзаполнение кода). `ctx.save`/achievements/records —
вне области (эфемерно).

**Тесты:**

- **Баланс** (`tests/lodge-balance.test.ts`): `createRun` детерминирован
  (один seed → идентичный `Run`); на seed, который раньше был однобоким (напр.
  2026), теперь обе роли встречаются как `lockOwner`; сметка по многим seed
  (каждый забег с ≥2 пазлами содержит обе роли как `lockOwner`); `validateRun
  === []`; цепочка решаема `solutionEvents`-прогоном.
- **Регистрация** (`tests/lodge-registration.test.ts`): форма
  `LODGE_DEFINITION` (id `lodge`, `theme 'shadow'`, `status 'available'`,
  непустые теги/таглайн/описание/bootHint).
- **Регрессия:** полный прогон (74) остаётся зелёным — баланс меняет владельцев
  только у однобоких забегов; сессионные/конвергентные тесты считают владельца
  динамически.
- **Финал:** `npm run typecheck` + `npm test` + `npm run build`; ручной smoke
  портала (каталог `/games` → запуск Ложи `/play/lodge` → лобби → создать/войти
  в двух вкладках → ESCAPED) — делегируется (нужен браузер), как smoke прошлых
  этапов.

## Секция 3 — Границы Этапа 3a (зафиксировано)

- 3D-комнаты подсказки (остаются clue-карты; отдельный под-этап).
- Таймер свечей, голос, пинги/маркеры (отдельные под-этапы).
- Новые архетипы пазлов (отдельный под-этап).

- Персистентность партий, `ctx.save`/achievements/records (эфемерно).
- Полная балансировка распределения замков (делаем минимум — гарантию ≥1 на
  роль; равномерный сплит — не цель, YAGNI).
- Соло как «настоящий» режим каталога (остаётся только на dev-роуте).
