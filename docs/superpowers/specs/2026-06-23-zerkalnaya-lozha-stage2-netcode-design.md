# Зеркальная Ложа — Этап 2 (реалтайм-неткод) дизайн

- **Дата:** 2026-06-23
- **Статус:** дизайн утверждён, готов к плану реализации
- **Область:** сетевой слой кооп-игры на двоих поверх движка Этапа 0 и
  стора/сцены Этапа 1. Транспорт-агностичное хост-авторитетное ядро + лобби по
  коду + три адаптера транспорта (in-memory / BroadcastChannel / Supabase
  Realtime). Реальная игра двух игроков (две вкладки или две машины).
- **Зависит от:** Этап 0 (движок `src/games/lodge/engine/`, PR #7) и Этап 1
  (стор/сцена `src/games/lodge/ui/`, PR #9). Этот этап стэкнут на Этап 1.

  неткод, и Секция 5 — «Этап 2»).

## 0. Контекст и решения

Движок Этапа 0 даёт чистый детерминированный API (`createRun`, `initRunState`,
`applyEvent`, `hashState`, `validateRun`). Этап 1 даёт `useLodgeStore`, чьё
`dispatch(puzzleId, by, event)` намеренно повторяет форму `LodgeEvent` — это шов
для сети. На текущей ветке **Supabase-клиента нет** (auth/cloud-интеграция живёт
на незамёрженном PR #5), `@supabase/supabase-js` не в зависимостях, сейвы
локальные.

Решения брейншторма (2026-06-23):

- **Транспорт-стратегия:** абстрактный `Transport`-интерфейс; ядро неткода
  тестируется без сети; два рантайм-адаптера — `BroadcastChannelTransport`
  (две вкладки, без сервера) и `SupabaseBroadcastTransport` (реальная сеть).
  Развязывает Этап 2 от PR #5.
- **Личность:** анонимно — эфемерный `clientId` + отображаемое имя, вход по
  коду комнаты, без логина. Ноль зависимости от авторизации.
- **Авторитет:** хост-авторитетный порядок событий по `seq`; гость шлёт интенты.
- **Экран по роли:** переиспользуем 3D-станции Этапа 1 для своих lock-пазлов;
  для clue-пазлов — read-only «карта-подсказка». 3D-комнаты подсказки — Этап 3.

## Дизайн-столпы

1. **Транспорт-агностичное ядро.** Вся логика сессии/авторитета зависит только
   от `Transport`-интерфейса; адаптеры (in-memory/BroadcastChannel/Supabase) —
   взаимозаменяемы. Тестируем ядро in-memory-хабом.
2. **Движок — источник истины, хост — источник порядка.** Состояние считает
   `applyEvent`; единый порядок задаёт хост через `seq`. Оба клиента применяют
   один поток → совпадают по построению.
3. **Минимальная правка Этапа 1.** Стор получает `applyServerEvent`/
   `setRunState`; станции зовут `act` из контекста (локально = `dispatch`,
   в сети = `submit`). Сцена не меняется.
4. **Самовосстановление.** `seq`-порядок ловит дубли/пропуски; `hashState`
   ловит десинк; снапшот чинит реконнект и десинк.
5. **Эфемерность.** Партии не персистятся; личности эфемерны; миграции хоста
   нет (осознанно вне Этапа 2).

## Секция 1 — Опыт и определение «готово»

Поток: игрок вводит имя → **Создать комнату** (6-символьный код, становится
хостом) или **Войти** по коду. Когда оба в комнате и «готовы», хост назначает
роли (хост = A, гость = B) и общий `seed`, рассылает `room` → оба строят один и
тот же `Run` и входят в игру.

Хост-авторитет: гость на клик шлёт `intent` хосту; хост валидирует против
каноничного состояния, присваивает `seq`, применяет `applyEvent`, рассылает
`event{lodgeEvent, stateHash}`. Хост свои действия проводит через ту же логику.
Оба применяют `event` по `seq` → состояния совпадают.

Экран по роли: у каждого игрока часть пазлов — его (он `lockOwner` → 3D-станция
Этапа 1), часть — партнёра (он `clueOwner` → read-only clue-карта). Вместе
покрывают все пазлы.

«Готово» для Этапа 2:

1. Два клиента (две вкладки через BroadcastChannel или две машины через
   Supabase) входят по коду, видят друг друга (presence) и роли.
2. Хост стартует — у обоих идентичный `Run`; каждый видит свои lock-станции и
   clue-карты партнёрских пазлов.
3. Действие одного появляется у обоих (через хост-авторитет); станции/состояние
   синхронны; решение всех пазлов → у обоих `ESCAPED`.
4. Реконнект: вышедший и вернувшийся клиент получает снапшот и догоняет;
   рассинхрон ловится `hashState` и чинится снапшотом.

## Секция 2 — Архитектура

Всё новое — под `src/games/lodge/net/`.

**`Transport` (интерфейс)** — единственная зависимость ядра от провода:
```ts
interface Transport {
  selfId: string;
  connect(): Promise<void>;
  disconnect(): void;
  send(msg: NetMessage): void;
  onMessage(cb: (m: NetMessage) => void): () => void;
  onPresence(cb: (peers: PeerInfo[]) => void): () => void;
}
```

**`NetMessage` (протокол):**
- `hello {from, name}` — гость объявляется.
- `room {seed, difficulty, roles, hostId}` — хост назначает seed+роли.
- `intent {from, puzzleId, event}` — гость → хост: «хочу сделать X».
- `event {lodgeEvent, stateHash}` — хост → всем: каноничное событие по `seq`.
- `snapshot {runState, lastSeq}` — хост → переподключившемуся: полное состояние.
- `resync {from}` — клиент → хост: «хеш разошёлся / пропуск seq, пришли снапшот».

**`LodgeSession`** (одна на игрока; держит `transport`, `role`, `isHost`,
`lastSeq`):
- Локальное действие: хост — через свою авторитет-логику; гость — `intent`.
- Авторитет (только хост): на `intent`/своё действие — валидирует, `seq =
  ++counter`, `applyEvent`, рассылает `event{…, stateHash}`.
- Приём `event`: оба применяют по `seq` в стор; сверяют `stateHash`; при
  несовпадении/пропуске — `resync`.
- Presence: новый пир → хост шлёт `room` + `snapshot`; уход пира → пауза.

**Шов со стором (минимальная правка Этапа 1):** `useLodgeStore` получает
`applyServerEvent(lodgeEvent)` и `setRunState(rs)`; локальный `dispatch`
остаётся для дев-режима. Станции зовут `act(puzzleId, by, event)` из контекста:
локально `act = store.dispatch`, в сети `act = session.submit`. Сцена не
меняется.

**Три адаптера** (все реализуют `Transport`):
- `InMemoryTransport` — общий in-process хаб; тесты/фикстуры.
- `BroadcastChannelTransport` — `BroadcastChannel('lodge:'+code)`, presence
  через хартбит; игра в двух вкладках без сервера.
- `SupabaseBroadcastTransport` — `createClient(url, anonKey)` из env
  `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY`, канал `lodge:<code>` (broadcast
  + presence). Новая зависимость `@supabase/supabase-js`. Нет env → адаптер
  недоступен, харнесс остаётся на BroadcastChannel.

**Файлы** `src/games/lodge/net/`: `types.ts` (Transport/NetMessage/PeerInfo),
`roomCode.ts` (генерация/валидация), `session.ts` (LodgeSession + авторитет),
`transports/{inMemory,broadcastChannel,supabase}.ts`, `index.ts`. Плюс правки
стора (`applyServerEvent`/`setRunState`) и UI: `dev/LobbyScreen.tsx`, сетевой
вариант харнесса, `ui/ClueCard.tsx`, выбор вида по роли.

## Секция 3 — Отказоустойчивость и тестирование

**Отказоустойчивость:**
- **Порядок/пропуски:** клиент держит `lastSeq`; `seq ≤ lastSeq` — дубль
  (игнор); `seq > lastSeq+1` — пропуск → `resync`, не применять наперёд.
- **Десинк:** `event` несёт `stateHash`; после применения клиент сверяет свой
  `hashState`; не сошлось → `resync` → снапшот.
- **Реконнект:** presence-возврат → хост шлёт `room` + `snapshot` → клиент
  `setRunState`, подтягивает `lastSeq`, догоняет.
- **Партнёр отвалился:** presence-drop → оверлей паузы у обоих, ввод на паузе.
- **Уход хоста:** сессия корректно завершается («хост покинул»); миграции нет.
- **Валидность интентов:** хост применяет `applyEvent`, который уже no-op для
  решённых/неизвестных пазлов и игнорирует чужие архетип-события.

**Тестирование** (vitest, без живого Supabase):
- **Сходимость:** `InMemoryTransport` + две `LodgeSession` через общий хаб;
  действия с обеих сторон → оба стора идентичны (`hashState` равны) и `escaped`.
- **Хост-авторитет/порядок:** одновременные интенты с двух сторон → единый
  детерминированный порядок по `seq` хоста.
- **Реконнект:** уронить гостя, продвинуть состояние, вернуть → снапшот
  приводит гостя к состоянию хоста.
- **Десинк:** впрыснуть расходящееся событие → `resync` → восстановление.
- **Роли/код:** хост = A, гость = B; `roomCode` генерится/валидируется
  (round-trip).
- **`BroadcastChannelTransport`** — лёгкий тест, если `BroadcastChannel` есть в
  среде; иначе помечаем и проверяем вручную в двух вкладках.
- **`SupabaseBroadcastTransport`** — тонкая обёртка над SDK, как R3F-вьюхи:
  юнит-тестом не покрываем; живая проверка — секретами пользователя
  (делегируется, как smoke Этапа 1).

**Фазовость** (план разобьёт): протокол/типы + `roomCode` → `InMemoryTransport`
→ `LodgeSession`+авторитет (ядро под тестами) → шов со стором → лобби+сетевой
харнесс+clue-карта → `BroadcastChannelTransport` → `SupabaseBroadcastTransport`.
