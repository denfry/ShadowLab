# Зеркальная Ложа — дизайн кооп-эскейпа в 3D (реалтайм)

- **Дата:** 2026-06-23
- **Статус:** дизайн утверждён, готов к плану реализации
- **Область:** новая игра портала `src/games/lodge` — реалтайм кооперативный
  эскейп на двоих в стилизованном 3D (React Three Fiber). Зеркальные комнаты,
  асимметричная информация, синхронизация через Supabase Realtime.
- **Рабочее название:** «Зеркальная Ложа» (placeholder; альтернативы — «Пока
  горят свечи», «Два крыла»).

---

## 0. Контекст и решения брейншторма

Портал уже содержит игры с чистыми детерминированными движками + тестами
(`shadow-trace`, `colony`, `among-the-quiet`), реестр `GameRegistry` с ленивой
загрузкой модулей, Supabase (авторизация + облачные сейвы) и стек
React 18 + Vite + Zustand + Tailwind. 3D-библиотеки пока нет — для браузера
естественный путь **React Three Fiber** (Three.js + R3F + drei).

Это первая реалтайм-мультиплеерная игра портала. По итогам брейншторма выбрано:

- **Формат совместной игры:** реалтайм вместе (живое присутствие), не асинхрон.
- **Концепт:** кооп-эскейп (а не социальная дедукция / prop hunt / пати-хаб).
- **Модель кооперации:** разделённые роли (вайб «Keep Talking and Nobody
  Explodes») — самый лёгкий неткод, максимум общения.
- **Топология:** зеркальные 3D-комнаты (вайб «We Were Here») — каждый игрок в
  своей 3D-комнате, перекрёстные головоломки, все честно «в 3D».
- **Контент:** параметризованные архетипы пазлов + seed-рандом (не ручная
  кампания, не полная процедурка) — детерминированный движок + тесты, высокая
  реиграбельность, ответы нельзя зашарить между компаниями.
- **Сеттинг:** оккультный особняк / тайное общество (руны, свечи, созвездия,
  алхимия) — богатейший «словарь» описываемых символов для «говорящих» пазлов.

**Дефолты (утверждены):** 2 игрока на старте (архитектурно закладываем «2 пары»
позже); тон расслабленный + мягкое давление (свечи-таймер на сессию, подсказки и
ретраи, без жёсткого фейла и софт-локов); сначала вертикальный срез, потом
расширение библиотеки архетипов; стилизованный low-poly; **без физ-движка**
(пазлы — конечные автоматы, не твёрдые тела); голос — внешний (Discord), голос мы
не делаем.

## Дизайн-столпы

1. **Асимметрия рождает разговор.** Ключ к ТВОЕМУ замку всегда в комнате
   напарника. Без описания вслух пройти нельзя — это и есть веселье.
2. **Детерминизм — фундамент синка.** Один seed → идентичный забег у обоих;
   общий поток событий + общий reducer → состояния совпадают по построению.
3. **«Честная» решаемость.** Валидатор гарантирует: каждый забег проходим, ни
   один замок не заперт в одной комнате со своей подсказкой, нет софт-локов.
4. **Лёгкий неткод.** По сети — только дискретные события состояния пазла, не
   движение аватаров. Дизайн терпит ~100–200 мс задержки.
5. **Логика в движке, UI тонкий.** Вся тяжёлая логика — в чистом TS-движке под
   тестами; R3F-слой только рисует и шлёт интенты.

## Секция 1 — Концепция и игровой цикл

Двое посвящённых заперты в разных крыльях оккультного особняка. Крылья —
зеркала друг друга: те же механизмы, но символы и коды, открывающие ТВОЙ
механизм, спрятаны в комнате напарника. Единственный выход — описывать друг
другу голосом, что видишь, и вместе проворачивать ритуальные пазлы, пока не
догорят свечи и не откроется центральный зал.

Игровой цикл забега:

1. **Лобби.** Хост создаёт комнату, получает короткий код; друг входит по коду;
   оба жмут «готов» → движок из общего seed детерминированно генерит забег → оба
   клиента строят свои зеркальные комнаты.
2. **В комнате.** Осматриваешься (клик-рейкаст по предметам, без физики),
   находишь элементы пазла и подсказки, предназначенные напарнику, говоришь по
   войсу, вводишь решения.
3. **Связка.** Решённый пазл шлёт событие состояния → в обеих комнатах
   открывается следующий механизм / дверь.
4. **Финал.** Цепочка из ~3 пазлов-архетипов → общий финальный замок → побег
   (до того, как догорят свечи).

## Секция 2 — Архитектура и детерминированный движок

Движок повторяет структуру `among-the-quiet/engine/`: чистый TS, без React/сети
внутри, всё покрывается vitest.

**Раскладка** `src/games/lodge/engine/`:

- `types.ts` — `Run`, `RoomView` (что видит игрок A / B), `PuzzleInstance`,
  `Archetype`, `LodgeEvent`, `RunState`.
- `seed.ts` — `mulberry32` (как в `colony`/`among-the-quiet`).
- `archetypes/` — реестр архетипов, по файлу на каждый.
- `generate.ts` — `createRun(seed, config) → Run`.
- `reducer.ts` — `applyEvent(state, event) → state'` (чистый, событийный).
- `validator.ts` — инвариант решаемости и асимметрии.
- `index.ts` — публичный barrel.

**Контракт архетипа.** Каждый архетип описывает, как из rng+параметров собрать
пару зеркальных видов и как проверять решение:

```ts
interface Archetype {
  id: string;
  generate(rng: Rng, difficulty: Difficulty): {
    a: RoomView;           // что показать/отрисовать игроку A
    b: RoomView;           // ...игроку B
    solution: Solution;    // канонический ответ
    clueOwner: 'A' | 'B';  // в чьей комнате лежит подсказка
    lockOwner: 'A' | 'B';  // в чьей комнате стоит замок (≠ clueOwner)
  };
  reduce(inst: PuzzleInstance, event: LodgeEvent): PuzzleInstance;
  isSolved(inst: PuzzleInstance): boolean;
}
```

Стартовые архетипы (оккультный словарь):

- **Диск ↔ Легенда.** У A — диск с алхимическими символами; у B — легенда
  «символ→цифра» и целевая последовательность. A крутит диск по диктовке B.
- **Созвездие.** У A — звёздная карта (какие звёзды соединить); у B — реальная
  пробивная панель-созвездие.
- **Свечи по порядку.** У A — стих с порядком зажигания; у B — канделябр,
  который зажигает в этом порядке.

**Контракт детерминизма и синка.** `createRun(seed)` чист → из одного seed оба
клиента строят идентичные `Run`. Игра — поток дискретных `LodgeEvent` (напр.
`{ type: 'dial.set', puzzleId, value }`, `{ type: 'candle.light', puzzleId, idx }`).
Оба клиента гоняют один и тот же `reducer` на одном и том же упорядоченном
потоке событий → состояния совпадают по построению (лёгкий lockstep; события
редкие, потому надёжно).

**Валидатор** (тестируется как `validateStation` в Among the Quiet): для каждого
пазла `lockOwner ≠ clueOwner`; ответ выводим из подсказок комнаты напарника; вся
цепочка проходима; нет недостижимых замков и софт-локов.

## Секция 3 — Неткод, рендер и обработка ошибок

**Неткод** `src/games/lodge/net/`:

- Канал Supabase Realtime на код комнаты, режим `broadcast` (эфемерно, без
  записи в БД).
- **Хост-авторитет по порядку событий.** Любое локальное действие игрока →
  интент хосту. Хост валидирует против каноничного состояния, присваивает
  монотонный `seq`, прогоняет `reducer`, рассылает обоим итоговый
  `LodgeEvent{seq, …}`. Оба применяют события строго по `seq` → единый порядок.
- В v1 без оптимистичного предсказания — события редкие, ~200 мс незаметны.
  Полезная нагрузка крошечная → латентность-терпимо.
- **Presence** Supabase — кто подключён, детект обрывов.

**Рендер** `src/games/lodge/ui/` (R3F, ленивая загрузка как Phaser сейчас):

- `LodgeScene.tsx` — `<Canvas>`, строит комнату игрока из `Run` + роли.
  Навигация упрощённая: клик-перемещение к точкам интереса (не свободный WASD) —
  меньше арта и коллизий, дружелюбно к мобилкам. Осмотр — рейкаст-клик (drei).
- Интерактивные компоненты (`Dial`, `ConstellationBoard`, `Candelabra`) читают
  состояние своего `inst` и шлют интенты.
- HUD поверх канваса (React+Tailwind): подсказка-цель, инструмент пинга/маркера,
  таймер свечей, статус связи.
- `definition.ts` + `LodgeGameModule.tsx` — регистрация в существующем
  `GameRegistry`. Мост состояние↔UI — Zustand.

**Обработка ошибок:**

- **Реконнект:** при возврате хост шлёт полный снапшот (`Run` + `RunState` +
  `lastSeq`) → клиент пересобирает сцену.
- **Защита от десинка:** каждое событие несёт `seq` + дешёвый `stateHash`; если
  после применения хеш не сошёлся — клиент запрашивает снапшот.
- **Партнёр отвалился:** presence-drop → оверлей «партнёр переподключается…»,
  таймер на паузе, продолжение по возврату.
- **Решаемость** гарантируется валидатором + эскалация подсказок против тупика.
- **Миграция хоста** — вне v1: если хост ушёл насовсем, сессия корректно
  завершается (известное ограничение; для двоих друзей приемлемо).

## Секция 4 — Тестирование

Vitest, как `aq-*` тесты. Тяжёлая логика — в чистом движке; UI тонкий.

- **Движок:**
  - *Детерминизм:* один seed → идентичный `Run` для обеих ролей; `reducer` на
    одинаковом потоке событий → идентичное состояние.
  - *Инвариант асимметрии:* `lockOwner ≠ clueOwner` у каждого пазла.
  - *Решаемость:* ответ выводим из подсказок комнаты напарника; вся цепочка
    проходима; нет недостижимых замков и софт-локов.
  - *По архетипу:* `generate` даёт валидные виды+ответ; переходы `reduce`;
    корректность `isSolved`; неверные ответы отвергаются.
  - *Стабильность хеша:* `stateHash` детерминирован.
- **Сеть** (симуляция, без реального Supabase): два in-memory клиента +
  фейковый хост-транспорт; гоняем интенты → проверяем сходимость (оба в
  идентичном состоянии); снапшот-реконнект == живое состояние; обработка
  переупорядоченных/потерянных событий.
- **UI:** лёгкие smoke-тесты компонентов.

## Секция 5 — Фазы реализации

Каждый этап — изолированный worktree + свой PR (как ведётся Among the Quiet).
Спека покрывает игру целиком; план реализации начинаем с Этапа 0.

- **Этап 0 — движок** (чистый, без UI/сети): seed, типы, 2–3 архетипа,
  `generate`, `reducer`, `validator` + полные тесты. Артефакт: зелёный прогон.
- **Этап 1 — вертикальный срез** (R3F, локально, 1 роль): рендер одной комнаты
  из `Run`, клик-интеракция, локальное решение одного архетипа, скелет HUD.
  Партнёра эмулируем дебаг-панелью. Доказываем 3D+интеракцию.
- **Этап 2 — сеть** (Supabase Realtime): лобби (создать/войти по коду), шеринг
  seed, хост-авторитетный синк событий, presence, реконнект/снапшот, защита от
  десинка. Полный цикл лобби→решение→побег вдвоём.
- **Этап 3 — контент и полиш:** больше архетипов, таймер свечей, пинг/маркер,
  звук/атмосфера, регистрация в каталоге портала, баланс.
